# Руководство по прошивке датчиков

**Привязка:** MASTER-SPEC.md v3.1
**Аудитория:** firmware-разработчик, интегратор

---

## Обзор

Датчик — автономное устройство (ESP32, nRF и т.п.), которое подключается к серверу по Wi-Fi через MQTT-брокер (Mosquitto). Датчик периодически отправляет телеметрию (температура, влажность, батарея) и поддерживает механизм онлайн/офлайн статуса через LWT.

Вся логика обработки — на стороне сервера. Датчик только:
1. Подключается к MQTT с credentials
2. Публикует online-статус
3. Отправляет телеметрию по таймеру
4. При отключении — брокер публикует LWT (offline)

---

## Типы датчиков

| Код | Название | Обязательные поля | Пример serial |
|---|---|---|---|
| TH | Temperature + Humidity | `t`, `h` | SENS-TH-00042 |
| TP | Temperature Probe | `t` | SENS-TP-00007 |
| T | Temperature Only | `t` | SENS-T-00001 |
| HM | Humidity Only | `h` | SENS-HM-00003 |

Тип определяется из serial-номера. Сервер проверяет, что датчик отправляет обязательные поля для своего типа. Если `SENS-TH-*` пришлёт payload без `t` или `h` — сообщение будет отклонено.

### Формат serial

```
SENS-{TYPE}-{NNNNN}
```

- `SENS` — фиксированный префикс
- `{TYPE}` — код типа: `TH`, `TP`, `T`, `HM` (1–2 заглавные латинские буквы)
- `{NNNNN}` — пятизначный номер с ведущими нулями (00001–99999)

Regex: `^SENS-[A-Z]{1,2}-\d{5}$`

Serial прошивается в устройство и не меняется. Он же используется как идентификатор в MQTT-топиках и API.

## Регистрация (provisioning)

Перед тем как датчик сможет подключиться, его нужно зарегистрировать на сервере. Это делает оператор через API или provision-cli.

### Что получает firmware-инженер после регистрации

| Параметр | Пример | Описание |
|---|---|---|
| MQTT Host | 192.168.1.50 | IP сервера в LAN |
| MQTT Port | 1883 | По умолчанию |
| Username | dev_sens_th_00042 | Уникальный, формат `dev_{serial_lowercase}` |
| Password | a7f3e9b1c4d2... | 32-символьная hex-строка, выдаётся один раз |
| Telemetry topic | d/SENS-TH-00042/t | Куда публиковать телеметрию |
| Status topic | d/SENS-TH-00042/s | Куда публиковать online/offline |

**Пароль возвращается один раз** при регистрации. Его нужно сохранить и прошить в устройство. Повторно получить пароль нельзя — только перерегистрировать (удалить + создать заново).

> **Формат пароля:** 32 символа lowercase hex (`[a-f0-9]{32}`), например `a7f3e9b1c4d2f5a8e2b6c9d3e7f0a4b8`. Генерируется сервером через CSPRNG.

### Как прошить credentials

Варианты (на усмотрение firmware-команды):
- Hardcode при компиляции (для прототипа)
- Через serial-консоль (AT-команды)
- Через BLE provisioning
- Через Wi-Fi AP (captive portal)
- Через NVS (non-volatile storage) на ESP32

Минимально нужно записать в устройство:
```
MQTT_HOST=192.168.1.50
MQTT_PORT=1883
MQTT_USERNAME=dev_sens_th_00042
MQTT_PASSWORD=a7f3e9b1c4d2...
DEVICE_SERIAL=SENS-TH-00042
```

## Подключение к MQTT

### Параметры подключения

| Параметр | Значение |
|---|---|
| Protocol | MQTT v3.1.1 или v5 |
| Transport | TCP (не WebSocket) |
| Port | 1883 |
| TLS | Нет в P1 (запланировано P2) |
| Username | из provisioning |
| Password | из provisioning (plaintext) |
| Client ID | любой уникальный, рекомендуется `SENS-TH-00042` или `sens_th_00042_{random}` |
| Keep Alive | 60 секунд (рекомендуется) |
| Clean Session | true |

### LWT (Last Will and Testament)

**Обязательно настроить при подключении:**

| Параметр | Значение |
|---|---|
| Will Topic | `d/{serial}/s` (например `d/SENS-TH-00042/s`) |
| Will Payload | `0` (ASCII, один символ) |
| Will Retain | `true` |
| Will QoS | `1` |

LWT срабатывает, когда брокер обнаруживает, что клиент отключился (потеря связи, crash, таймаут keep-alive). Брокер автоматически публикует `0` в status-топик — сервер отмечает датчик как offline.

### Сразу после подключения

Опубликовать online-статус:

| Параметр | Значение |
|---|---|
| Topic | `d/{serial}/s` |
| Payload | `1` (ASCII, один символ) |
| Retain | `true` |
| QoS | `1` |

Это перезаписывает retained-значение `0` от предыдущего LWT, и сервер отмечает датчик как online.

## Формат телеметрии

### Топик

```
d/{serial}/t
```

Пример: `d/SENS-TH-00042/t`

QoS: `1` (at least once). Retain: `false`.

### Payload (JSON)

```json
{
  "v": 1,
  "id": "SENS-TH-00042",
  "ts": 1752588600,
  "mid": "a1b2c3d4",
  "t": -18.3,
  "h": 45.2,
  "bat": 87,
  "rssi": -67,
  "fw": "0.1.3"
}
```

### Описание полей

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `v` | integer | да | Версия схемы. Всегда `1` |
| `id` | string | да | Serial устройства, должен совпадать с зарегистрированным |
| `ts` | integer | да | Unix timestamp в секундах (UTC). `time()` на устройстве |
| `mid` | string | да | Message ID, уникальный в пределах устройства (см. ниже) |
| `t` | number | зависит от типа | Температура °C. Диапазон: -55 .. +125 |
| `h` | number | зависит от типа | Влажность %. Диапазон: 0 .. 100 |
| `bat` | integer | нет | Заряд батареи %. Диапазон: 0 .. 100. Не отправлять если wired |
| `rssi` | integer | нет | Wi-Fi RSSI в dBm. Диапазон: -120 .. 0 |
| `fw` | string | нет | Версия прошивки. Макс 20 символов |
| `up` | integer | нет | Uptime в секундах с момента загрузки |

### Обязательность полей по типу датчика

| Тип | `t` | `h` | `bat` |
|---|---|---|---|
| TH | обязательно | обязательно | если battery |
| TP | обязательно | не отправлять | обычно нет (wired) |
| T | обязательно | не отправлять | если battery |
| HM | не отправлять | обязательно | если battery |

Если обязательное поле отсутствует — сервер отклоняет сообщение и пишет в audit log `payload.missing_capability`.

Если необязательное поле присутствует — сервер его игнорирует (не ошибка).

### Ограничение размера

Максимум **512 байт** JSON. Сообщения больше отклоняются.

При типичном payload (~120 байт) это не проблема. Но не добавляйте лишних полей.

## Message ID (mid)

`mid` используется для дедупликации на сервере. Сервер хранит пары `(device_id, message_id)` и отклоняет дубли без ошибки.

### Требования

- Уникален в пределах одного устройства (не глобально)
- Строка от 1 до 64 символов
- Не обязан быть числовым

### Рекомендуемые стратегии генерации

**Вариант 1: Monotonic counter (рекомендуется для микроконтроллеров)**

```c
static uint32_t msg_counter = 0;
// Сохранять в NVS при перезагрузке, или добавить случайный сдвиг
char mid[16];
snprintf(mid, sizeof(mid), "%u", ++msg_counter);
```

Плюсы: минимальный расход памяти, гарантированная уникальность при одном потоке.
Минусы: после перезагрузки счётчик сбрасывается — добавьте NVS-сохранение или случайный начальный сдвиг.

**Вариант 2: Timestamp + random suffix**

```c
char mid[32];
snprintf(mid, sizeof(mid), "%lu_%04x", time(NULL), esp_random() & 0xFFFF);
```

**Вариант 3: UUID-like**

```c
// Если есть crypto RNG
uint8_t buf[8];
esp_fill_random(buf, 8);
// hex-encode → 16 символов
```

## Интервал отправки

| Параметр | Рекомендация |
|---|---|
| Нормальный режим | 60 секунд |
| Минимальный интервал | 10 секунд (не чаще) |
| Таймаут офлайна на сервере | 300 секунд (настраивается, env `DEVICE_OFFLINE_TIMEOUT_SEC`) |

Если датчик не отправлял телеметрию и LWT не сработал более 5 минут — сервер отметит его как offline по таймауту.

## ACL (Access Control)

Датчик имеет право:

| Действие | Топик |
|---|---|
| Publish (write) | `d/{serial}/t` |
| Publish (write) | `d/{serial}/s` |

Устройство **не должно подписываться** на какие-либо топики — любая подписка будет отклонена ACL.

Попытка публикации в чужой топик (например `d/SENS-TH-00002/t` от `SENS-TH-00001`) будет отклонена брокером.

## Алгоритм работы датчика (псевдокод)

```
1. Загрузка
   ├── Прочитать из NVS: MQTT_HOST, MQTT_PORT, USERNAME, PASSWORD, SERIAL
   ├── Подключиться к Wi-Fi
   └── Если нет credentials → войти в режим настройки (AP/BLE)

2. Подключение к MQTT
   ├── Настроить LWT: topic="d/{SERIAL}/s", payload="0", retain=true, qos=1
   ├── Подключиться к брокеру
   ├── При успехе → опубликовать "1" в "d/{SERIAL}/s" (retain, qos 1)
   └── При ошибке → подождать, переподключиться

3. Основной цикл (каждые 60 секунд)
   ├── Прочитать сенсоры (температура, влажность)
   ├── Прочитать батарею, RSSI
   ├── Сформировать JSON payload:
   │     {
   │       "v": 1,
   │       "id": SERIAL,
   │       "ts": unix_time(),
   │       "mid": next_message_id(),
   │       "t": temperature,
   │       "h": humidity,     // если тип поддерживает
   │       "bat": battery,    // если батарейный
   │       "rssi": wifi_rssi,
   │       "fw": FIRMWARE_VERSION
   │     }
   ├── Опубликовать в "d/{SERIAL}/t" (qos 1, retain false)
   └── Deep sleep или delay

4. Отключение (crash / потеря Wi-Fi / deep sleep без disconnect)
   └── Брокер публикует LWT: "0" в "d/{SERIAL}/s"

5. Переподключение
   ├── MQTT-библиотека автоматически переподключается
   └── После reconnect → снова опубликовать "1" в status topic
```

## Примеры payload

### Датчик TH (температура + влажность, батарейный)

```json
{"v":1,"id":"SENS-TH-00042","ts":1752588600,"mid":"a1b2c3d4","t":-18.3,"h":45.2,"bat":87,"rssi":-67,"fw":"0.1.3"}
```

### Датчик TP (зонд температуры, от сети)

```json
{"v":1,"id":"SENS-TP-00007","ts":1752588600,"mid":"e5f6a7b8","t":4.1,"rssi":-52,"fw":"0.2.0","up":86400}
```

### Датчик T (только температура, батарейный)

```json
{"v":1,"id":"SENS-T-00001","ts":1752588600,"mid":"42","t":-20.1,"bat":63,"rssi":-71,"fw":"0.1.0"}
```

### Датчик HM (только влажность)

```json
{"v":1,"id":"SENS-HM-00003","ts":1752588600,"mid":"107","h":72.4,"bat":91,"rssi":-55,"fw":"0.1.0"}
```

### Минимальный валидный payload (датчик T)

```json
{"v":1,"id":"SENS-T-00001","ts":1752588600,"mid":"1","t":-20.0}
```

## Калибровка

На сервере для каждого устройства можно задать `calibrationOffsetC` (поправка в °C). Сервер прибавляет offset к значению `t` из payload перед записью:

```
readings.temperatureC = payload.t + device.calibrationOffsetC
```

Оригинальный JSON сохраняется в `rawPayload` без изменений.

Калибровка настраивается оператором через API (`PATCH /api/v1/devices/:serial`), датчику об этом знать не нужно — всегда отправляйте сырые показания сенсора.

## Wi-Fi

### Требования

- 2.4 GHz (стандарт для IoT-устройств)
- WPA2-PSK
- Датчик и сервер должны быть в одной LAN (или маршрутизируемой подсети)

### Рекомендации

- Сохранять Wi-Fi credentials в NVS
- При невозможности подключения — войти в режим настройки (AP mode или BLE)
- Retry с exponential backoff при потере связи
- При успешном Wi-Fi → подключаться к MQTT

---

## Синхронизация времени

Поле `ts` — Unix timestamp в секундах (UTC). Сервер не корректирует время.

### Рекомендации

- Использовать NTP при старте (pool.ntp.org или локальный NTP)
- Если NTP недоступен — использовать RTC (если есть) или uptime от загрузки + приблизительное начальное время
- Точность ±5 секунд достаточна для P1
- Не использовать локальное время — только UTC

## Энергосбережение (для батарейных датчиков)

| Стратегия | Описание |
|---|---|
| Deep sleep | Между отправками уходить в deep sleep. При пробуждении — Wi-Fi + MQTT connect + publish + disconnect + sleep. LWT сработает при disconnect |
| Light sleep | Поддерживать Wi-Fi, спать между публикациями. Keep-alive не прерывается |
| Always on | Для wired датчиков. Постоянное подключение, keep-alive 60s |

При deep sleep каждое пробуждение — это новое MQTT-подключение. Нужно:
1. Подключиться (с LWT)
2. Опубликовать `1` в status topic
3. Опубликовать telemetry
4. Disconnect:
   - **Graceful stop**: явно опубликовать `0` в status topic, затем disconnect. Это explicit offline, не LWT.
   - **Deep sleep без disconnect**: соединение обрывается — брокер публикует LWT (`0`).

> **Два пути offline:** Graceful shutdown = firmware сам публикует `0`. Аварийный обрыв = broker LWT. Сервер дополнительно считает устройство offline по таймауту (`DEVICE_OFFLINE_TIMEOUT_SEC`), если ни один из этих сигналов не поступил.

## Тестирование прошивки

### Проверка подключения

```bash
# Мониторинг топиков устройства (используйте admin-credentials, не device)
# Device credentials НЕ имеют права на подписку (ACL: publish-only)
mosquitto_sub -h 192.168.1.50 -p 1883 \
  -u admin -P '$MQTT_ADMIN_PASSWORD' \
  -t 'd/SENS-TH-00042/#' -v
```

### Проверка публикации

```bash
mosquitto_pub -h 192.168.1.50 -p 1883 \
  -u dev_sens_th_00042 -P 'пароль' \
  -t 'd/SENS-TH-00042/t' \
  -m '{"v":1,"id":"SENS-TH-00042","ts":1752588600,"mid":"test1","t":-18.3,"h":45.0,"bat":90,"rssi":-60,"fw":"0.1.0"}'
```

Проверить на сервере:

```bash
curl -H "Authorization: Bearer $API_TOKEN" \
  http://localhost:8080/api/v1/devices/SENS-TH-00042/readings?limit=1
```

### Проверка offline: graceful stop (explicit publish)

1. Запустить симулятор / подключить датчик
2. Остановить штатно (Ctrl+C / graceful shutdown)
3. Датчик сам публикует `0` в status topic перед disconnect
4. Проверить сразу:

```bash
curl -H "Authorization: Bearer $API_TOKEN" \
  http://localhost:8080/api/v1/devices/SENS-TH-00042
# connectivityStatus должен быть "offline"
```

### Проверка offline: LWT (аварийный обрыв)

1. Подключить датчик / запустить симулятор
2. Аварийно завершить процесс: `kill -9 <PID>`, выдернуть питание, разорвать сеть
3. Через ~90 секунд (1.5 × keep-alive) брокер опубликует LWT
4. Проверить:

```bash
curl -H "Authorization: Bearer $API_TOKEN" \
  http://localhost:8080/api/v1/devices/SENS-TH-00042
# connectivityStatus должен быть "offline"
```

### Проверка ACL

```bash
# Попытка публикации в чужой топик — должна быть отклонена
mosquitto_pub -h 192.168.1.50 -p 1883 \
  -u dev_sens_th_00042 -P 'пароль' \
  -t 'd/SENS-TH-00002/t' \
  -m '{"v":1,"id":"SENS-TH-00002","ts":1752588600,"mid":"hack","t":0}'
# Брокер отклонит — в логах Mosquitto будет "not authorised"
```

### Использование симулятора как reference

Симулятор (`tools/simulator`) реализует полный контракт датчика и может служить эталоном:

```bash
# Посмотреть что отправляет симулятор
DEVICE_SERIAL=SENS-TH-00042 \
MQTT_URL=mqtt://localhost:1883 \
MQTT_USERNAME=dev_sens_th_00042 \
MQTT_PASSWORD=пароль \
INTERVAL_SEC=5 \
pnpm --filter @sensor/simulator simulate
```

Исходный код: `tools/simulator/src/index.ts` — reference-реализация LWT, online publish, payload format.

## Checklist готовности прошивки

```
[ ] Serial прошит в формате SENS-{TYPE}-{NNNNN}
[ ] Wi-Fi подключение с retry
[ ] NTP-синхронизация (или RTC)
[ ] MQTT подключение с credentials из provisioning
[ ] LWT настроен: topic d/{serial}/s, payload "0", retain true, QoS 1
[ ] При connect: publish "1" в d/{serial}/s, retain true, QoS 1
[ ] Telemetry по таймеру в d/{serial}/t, QoS 1, retain false
[ ] Payload соответствует DevicePayloadSchema (v=1, id, ts, mid + обязательные поля типа)
[ ] mid уникален (counter в NVS или random)
[ ] Payload ≤ 512 байт
[ ] Обязательные поля для типа присутствуют (t для TH/TP/T, h для TH/HM)
[ ] bat только для батарейных датчиков
[ ] Reconnect при потере связи
[ ] Graceful shutdown (если применимо)
```

## Частые ошибки

| Проблема | Причина | Решение |
|---|---|---|
| Connection refused | Неверные credentials | Проверить username/password из provisioning |
| Сообщения не появляются в readings | payload не проходит валидацию | Проверить формат JSON, `v` должен быть `1` (число), serial в `id` должен совпадать с зарегистрированным |
| `payload.missing_capability` в audit | Отсутствует обязательное поле | TH должен отправлять и `t`, и `h` |
| `device.unknown_message` в audit | Датчик не зарегистрирован | Сначала provision через API |
| Датчик offline хотя работает | LWT/keep-alive не настроен, или интервал > 5 мин | Настроить LWT, уменьшить интервал или увеличить DEVICE_OFFLINE_TIMEOUT_SEC |
| ACL reject | Публикация в чужой топик | Проверить, что serial в топике совпадает с serial в credentials |
| Дубли игнорируются | Повторный `mid` | Убедиться что mid инкрементируется или рандомный |
